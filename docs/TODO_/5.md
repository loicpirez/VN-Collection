Audit findings — vndb-collection-new performance & DB hygiene

Severity scale: P0 = hot-path 10×+ regression candidate, P1 = hot-path but bounded, P2 = correctness/cleanup, P3 = nit. File paths absolute. Library size assumption: ~1000 in-collection VNs.

P0 — /api/collection does O(N) cached-release scans per request

src/app/api/collection/route.ts:97-118 — when the request carries any aspect filter OR ?group=aspect:

for (const id of allVnIds) {
  if (/^v\d+$/.test(id)) {
    materializeReleaseAspectsForVn(id);
    materializeReleaseMetaForVn(id);
  }
}

materializeReleaseAspectsForVn (src/lib/db.ts:3112) short-circuits when the VN already has a non-unknown signal — fine.materializeReleaseMetaForVn (src/lib/db.ts:3243) has no short-circuit. Per call it runs:

SELECT body FROM vndb_cache WHERE cache_key LIKE '% /release|%' LIMIT 200

— a full-scan LIKE with a leading wildcard (no index can serve it). Then it walks up to 200 cached payloads, JSON-parses each, iterates results[], filters by vns[].id === vnId, and re-upserts every linked row. Per-VN cost is proportional to the entire /release cache, not the VN's own releases.

For a 1000-VN library with the aspect filter active:

1000 × full-scan of vndb_cache LIKE filter

1000 × up to 200 JSON.parse calls of POST /release bodies (each can be 50-500 KB)

1000 × per-release upsert into release_meta_cache

1000 × per-VN UPDATE owned_release SET owned_platform = ... at line 3348-3365 (auto-fills singleton platforms)

Even worst case at 100 cached release pages × 50 releases each = 5,000,000 JSON inspections per request. This dwarfs every other cost in the audit.

Fix:

Same short-circuit pattern as materializeReleaseAspectsForVn: skip when release_meta_cache already has rows for this vn_id (use the existing idx_release_meta_cache_vn index) AND the rows are not older than the latest matching vndb_cache row.

Better: invert the loop — read every cached POST /release body once, iterate results[], dispatch to the right VN id from vns[], batch the upsert in a single transaction. One scan instead of N.

The LIKE '% /release|%' pattern is fragile (depends on __pathTag shape with a leading space). Replace with cache_key LIKE 'POST /release|%' (anchored prefix → uses the PRIMARY KEY index).

Same applies to materializeReleaseAspectsForVn:3124 — same LIKE pattern.

Note the file's own comment at line 124 acknowledges "the cache table is small and indexed by key" — that's true only for prefix LIKE; the leading-wildcard pattern there negates the claim.

Fix: cache the column set per table at the top of open():

const tableCols = new Map<string, Set<string>>();
function cols(table: string): Set<string> {
  if (!tableCols.has(table)) {
    const rows = db.prepare(`PRAGMA table_info(${table})`).all() as ColInfo[];
    tableCols.set(table, new Set(rows.map(r => r.name)));
  }
  return tableCols.get(table)!;
}
function ensureColumn(table: string, column: string, ddl: string): void {
  if (!cols(table).has(column)) {
    db.exec(`ALTER TABLE ${table} ADD COLUMN ${column} ${ddl}`);
    cols(table).add(column);
  }
}

UI / UX / Accessibility Audit — vndb-collection-new

Summary

The new repo is largely well-built and inherits the accessibility infrastructure of the previous version (Dialog shell, tap-target classes, skeleton primitives, i18n dictionary, status icons). However, a number of regressions have crept in — particularly around tap targets on icon-only buttons, raw enum bleed in user-facing text, missing ARIA wiring in dropdown menus, missing safe-area / fade-hint on mobile, unrotating chevrons on <details> summaries, and several Unicode glyphs that should be Lucide icons.

Severity legend:

CRITICAL — keyboard/touch users blocked or misled; clear violation of stated project conventions.

HIGH — measurable a11y deficit (missing labels, weak landmark wiring).

MEDIUM — UX rough edges (no skeleton, missing fade hint, missing icon, hardcoded enum strings).

LOW — style nit / convention inconsistency.
