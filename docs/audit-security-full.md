# Full Security Audit — VNDB Collection (`/Users/loicpirez/VNDB`)

Audit date: 2026-05-27
Scope: every API route under `src/app/api/**/route.ts` and every server lib under `src/lib/*.ts`.

Context: this is a single-user, self-hosted application with a strong baseline. `requireLocalhostOrToken` is applied to virtually every mutating route, `csrfGuard` (Next middleware in `src/proxy.ts`) covers all `/api/*` for state-changing methods, the SSRF allowlist (`isAllowedHttpTarget`) is enforced everywhere user-influenced URLs leave the box, SQL is parameterised everywhere, and JSON bodies go through `readJsonObject` then narrow each field with `typeof`. The findings below are deviations from those patterns or defence-in-depth gaps. Severity is calibrated against the loopback-only threat model — most issues are LOW or MEDIUM because the gate already reduces blast radius.

## Summary

- **CRITICAL**: 0
- **HIGH**: 0
- **MEDIUM**: 17 (info-leak via error messages on five routes; missing length caps on user-supplied strings; missing type/shape checks on color/icon/proxy fields; missing validation on the kobe `code` URL segment)
- **LOW**: 84
- **INFORMATIONAL**: 10

The application has a strong security baseline:
- Every mutating route is gated by `requireLocalhostOrToken`
- CSRF middleware covers all `/api/*` state-changing methods
- Outbound HTTP uses `isAllowedHttpTarget` everywhere user-influenced URLs flow
- DNS-rebinding defence on user-configured mirror URL
- All SQL is parameterised (no template-string interpolation of user data into `db.prepare()`)
- File-upload routes sniff magic bytes
- Content-Length pre-check before buffering multipart uploads
- Path-traversal blocked at file-route boundary AND in `lib/files.ts`
- `upstreamError` helper hides raw upstream error bodies from clients (but not used consistently — S-001 through S-005)
- VNDB throttle + circuit breaker prevent rate-limit abuse
- Constant-time admin-token comparison
- Proxy credentials never logged or returned (saveProxyConfig preserves stored password on empty/mask)

Recurring themes:
1. **Defensive error responses are inconsistent.** A handful of routes still ship raw `(e as Error).message` directly into JSON — see S-001, S-002, S-003, S-004 — even though `upstreamError` exists in `lib/api-error.ts` precisely for this case. Migrating them is a 5-line fix per file.
2. **String length caps are not universal.** Most routes apply `.slice(0, N)`. The exceptions cluster around `notes`, `description`, `edition_label`, and proxy credentials — see S-006 through S-013, S-015, S-051, S-052, S-063, S-064.
3. **Several read endpoints are intentionally unauthenticated** (CLAUDE.md documents this design choice). They expose PII at the LAN level when the operator runs outside loopback. S-095 through S-104 catalog the routes; the design tradeoff is intentional but should be a knob, not a hardcoded behaviour.
4. **Kobe `code` URL parameter unvalidated** (S-036) — the only mutating route in the audit where a path segment isn't shape-checked. Fix is a 1-line regex.
5. **`ensureColumn` interpolates identifiers** (S-060) — no current vulnerability because all call sites pass internal constants, but a `verifyIdentifier` helper would harden the pattern against future refactors.

Most LOW findings are accepted-risk for a single-user self-hosted app or fall into the "defence-in-depth — no concrete attack today" bucket. The 17 MEDIUM findings represent the actionable surface; they're concentrated in a small number of files and are all 1-5 line fixes.
